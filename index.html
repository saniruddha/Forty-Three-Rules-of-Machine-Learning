Evaluating Machine Learning Projects: Forty Three Rules of Machine Learning

ITCS 6155 - Knowledge-Based Systems - Project Group 14 - (University of North Carolina at Charlotte)

With the acceptance and utilization of the machine learning field today, many organizations leverage the power of machine learning for predictions, recommendations, evaluation of the performance of their products, etc. Are they all successful in developing a robust and unbiased machine learning solution for their organization?

Not many! Even organizations like Google have made mistakes in the past before leading the world in ways to implement and leverage the power of machine learning. They have shared their invaluable experience in the form of ‘Forty Three Rules of Machine Learning’ to help others develop great machine learning solutions. We have carefully referred to the information shared by Google to develop a good machine learning solution for our project on ‘Prediction of COVID-19 Cases’.

Before Machine Learning:
  • Rule #1: Don’t be afraid to launch a product without machine learning.

We have carefully checked the usability of our dataset which consists of 500,000 records and 11 features. Therefore, using machine learning will give us a 100% boost in making predictions instead of a heuristic approach.

  • Rule #2: First, Design and implement metrics.

We have come up with some strong metrics like ‘impact of ship entering a particular state’, ‘impact on individual states’, ‘likelihood of spread in neighboring states’, ‘trend in increase in the number of cases’ after carefully assessing the data quality for making predictions.

ML Phase I: Your First Pipeline:
  • Rule #4: Keep the first model simple and get the infrastructure right.

We are working on a classic regression problem in our project i.e. predicting the number of confirmed Coronavirus cases on a given date. Therefore, we have decided to keep it simple at first by properly implementing a basic linear regression model on simple features to perform predictions and later extend it to a more complex ML model.

  • Rule #14: Starting with an interpretable model makes debugging easier.

As we have decided to start implementation with a linear regression model, which is directly motivated by a probabilistic model, it will be easy to inspect the production system and identify problems.

ML Phase II: Feature Engineering:
  • Rule #16: Plan to launch and iterate.

We are building a system which will simplify incorporating new changes and model tuning that may arise as a future requirement for improvising our machine learning solution.

  • Rule #33: If you produce a model based on the data until January 5th, test the model on the data from January 6th and after.

To make sure to better understand the performance of the developed machine learning model, we will use this approach of testing the model on data with dates after the date in the training data.

ML Phase III: Slowed Growth, Optimization Refinement, and Complex Models:
  • Rule #40: Keep ensembles simple.

We are planning to implement predictions with the ensemble of basic linear regression models by making sure the increase in predicted probability of the underlying classifier does not decrease the predicted probability of ensembles.
